title: "Clause blocks: FROM, JOIN, WHERE, GROUP BY, ORDER BY"
description: "Reference for SQLa's clause blocks: FROM with JOIN support, WHERE comparisons, membership and quantified conditions, GROUP BY, HAVING, and ORDER BY."
---

Clause blocks attach below the SELECT block to build the rest of a SQL query. Each clause is its own puzzle piece with typed connectors that only accept the correct child blocks, guiding you toward syntactically valid SQL.

## Overview

Clause blocks use two distinct colors: orange/amber (hue 30) for filtering blocks (WHERE, comparison conditions) and teal/cyan (hue 160–230) for structural clauses (FROM, GROUP BY, ORDER BY, HAVING). All clause blocks use `SQL_STATEMENT` connections so they chain vertically in sequence.

<AccordionGroup>

<Accordion title="FROM simple block (sql_from_simple)">

The basic FROM clause. Specifies a single source table with no JOIN support.

**Fields:**
- `TABLE_NAME` — text input, default `mi_tabla`

**Connections:**
- `previousStatement` / `nextStatement`: `SQL_STATEMENT`

**Context menu:** Right-click the block and choose **Add JOIN clause** to automatically replace this block with the JOIN-capable `sql_from` block, preserving the table name.

**Generator output:**

```sql
FROM empleados
```

</Accordion>

<Accordion title="FROM block with JOINs (sql_from)">

The full FROM clause that supports JOIN blocks nested inside it. Created by right-clicking a simple FROM block and choosing **Add JOIN clause**, or by dragging it directly from the toolbox.

**Fields:**
- `TABLE_NAME` — text input, default `mi_tabla`

**Connections:**
- `previousStatement` / `nextStatement`: `SQL_STATEMENT`
- `JOINS` statement input: accepts a stack of `SQL_JOIN` blocks

**Generator output:**

```sql
-- FROM with no JOINs
FROM empleados

-- FROM with joins (join lines generated by JOIN blocks)
FROM empleados
INNER JOIN departamentos ON empleados.dept_id = departamentos.id
LEFT JOIN ubicaciones ON departamentos.ubicacion_id = ubicaciones.id
```

</Accordion>

<Accordion title="JOIN block (sql_join)">

Joins a second table to the query using a configurable join type and an `ON` condition. JOIN blocks stack vertically inside the `JOINS` input of the FROM block.

**Fields:**
- `JOIN_TYPE` — dropdown: `INNER`, `LEFT`, `RIGHT`, `FULL OUTER`
- `JOIN_TABLE` — text input, default `otra_tabla`

**Connections:**
- `previousStatement` / `nextStatement`: `SQL_JOIN` — chains inside FROM's `JOINS` slot
- `ON_CONDITION` value input: accepts `Condition` blocks (Comparison, Membership, Quantified Comparison)

**Auto-insert on creation:** When a JOIN block is placed on the workspace, its `onchange` handler automatically creates and connects a Comparison block with two `sql_expression_single` blocks (pre-filled with `tabla.columna`) into the `ON_CONDITION` input. This gives you a ready-to-edit ON condition immediately.

**Generator output:**

```sql
INNER JOIN departamentos ON empleados.dept_id = departamentos.id

LEFT JOIN proyectos ON empleados.id = proyectos.lider_id

RIGHT JOIN contratos ON contratos.empleado_id = empleados.id

FULL OUTER JOIN historial ON historial.empleado_id = empleados.id
```

If no condition block is connected, the generator emits a `/* condición ON requerida */` placeholder so you can still see the SQL structure.

</Accordion>

<Accordion title="WHERE block (sql_where)">

Filters rows using a condition expression. Accepts one condition block in its `CONDITION` input.

**Connections:**
- `previousStatement` / `nextStatement`: `SQL_STATEMENT`
- `CONDITION` value input: accepts `Condition` blocks

**Context menu:** Right-click to insert a condition block (Comparison, Membership, or Quantified Comparison).

**Generator output:**

```sql
WHERE salario > 50000

WHERE departamento = 'Ingeniería'
```

If the condition input is empty, the WHERE block generates no output at all (empty string) rather than invalid SQL.

</Accordion>

<Accordion title="Comparison block (sql_comparison)">

The most common condition type. Compares two values using a standard operator. Used inside WHERE, JOIN ON conditions, and HAVING.

**Fields:**
- `OPERATOR` — dropdown: `=`, `<>`, `!=`, `<`, `>`, `<=`, `>=`

**Connections:**
- `output`: `Condition`
- `LEFT` value input: accepts `Expression`, `ExpressionSingle`, `Column`, `Aggregate`
- `RIGHT` value input: accepts `Expression`, `ExpressionSingle`, `Column`

**Generator output:**

```sql
salario > 50000

email = 'usuario@empresa.com'

edad <= 65
```

<Note>
  The `ExpressionSingle` connection type is used for the inline `tabla.columna` notation inside JOIN ON conditions. This lets you type `empleados.dept_id` directly into the comparison operand fields.
</Note>

</Accordion>

<Accordion title="Membership block (sql_membership)">

Tests whether a value belongs to a set. Generates `IN` or `NOT IN` SQL.

**Fields:**
- `OPERATOR` — dropdown: `IN`, `NOT IN`

**Connections:**
- `output`: `Condition`
- `LEFT` value input: accepts `Expression`, `Column`, `Aggregate`
- `RIGHT` value input: accepts `Expression`, `Column` (the list of values)

**Generator output:** The right operand is automatically wrapped in parentheses.

```sql
departamento IN (ventas)

estado NOT IN (inactivo)
```

</Accordion>

<Accordion title="Quantified Comparison block (sql_quantified_comparison)">

Compares a value against all or any members of a subquery or list, using `ALL`, `ANY`, or `SOME` quantifiers.

**Fields:**
- `OPERATOR` — dropdown: `=`, `<>`, `!=`, `<`, `>`, `<=`, `>=`
- `QUANTIFIER` — dropdown: `ALL`, `ANY`, `SOME`

**Connections:**
- `output`: `Condition`
- `LEFT` value input: accepts `Expression`, `Column`, `Aggregate`
- `RIGHT` value input: accepts `Expression`, `Column`

**Generator output:** The right operand is wrapped in parentheses.

```sql
salario > ALL (50000)

edad = ANY (25)
```

</Accordion>

<Accordion title="GROUP BY block (sql_group_by)">

Groups result rows by one or more column values. Must appear after FROM and WHERE in the query chain. Aggregates in the SELECT list operate on each group independently.

**Connections:**
- `previousStatement` / `nextStatement`: `SQL_STATEMENT`
- `COLUMNS` value input: accepts `GroupByColumn` blocks

**Auto-insert on creation:** When placed on the workspace, a `sql_groupby_column` block is automatically connected into the `COLUMNS` input as a starting point.

**Generator output:**

```sql
GROUP BY departamento

GROUP BY departamento, cargo
```

If the COLUMNS input is empty, the block generates no output.

</Accordion>

<Accordion title="GROUP BY Column block (sql_groupby_column)">

An individual column reference inside a GROUP BY block. Chain multiple column blocks for multi-column grouping.

**Generator output:** Column names are comma-separated when chained.

```sql
-- Two groupby_column blocks chained
departamento, cargo
```

</Accordion>

<Accordion title="ORDER BY block (sql_order_by)">

Sorts the final result set by one or more columns, each with an optional direction.

**Connections:**
- `previousStatement` / `nextStatement`: `SQL_STATEMENT`
- `EXPRESSION` value input: accepts `Expression`, `Column`

**Direction per column:** When an Expression block is connected inside ORDER BY, the `order_by_expression_extension` adds a `DIRECTION` dropdown directly to that expression block (values: `ASC`, `DESC`). The direction is baked into the expression's generated code.

**Generator output:**

```sql
ORDER BY salario DESC

ORDER BY apellido ASC, nombre ASC
```

If the EXPRESSION input is empty, the block generates no output.

</Accordion>

<Accordion title="HAVING block (sql_having)">

Filters grouped results after GROUP BY aggregation. Accepts conditions that reference aggregate function blocks.

**Connections:**
- `previousStatement` / `nextStatement`: `SQL_STATEMENT`
- `EXPRESSION` value input: accepts `Expression`, `Column`, `Aggregate`, `Condition`

**Generator output:**

```sql
HAVING SUM(salario) > 100000

HAVING COUNT(id) >= 5
```

<Tip>
  For HAVING conditions, use the `_having` variants of aggregate blocks (e.g., `sql_sum_having`, `sql_count_having`). These blocks are closed (no `NEXT` input) and are designed to be placed directly inside a Comparison block that is then connected to HAVING's expression input.
</Tip>

</Accordion>

</AccordionGroup>

## Full clause chain example

The following query uses SELECT, FROM with INNER JOIN, WHERE, GROUP BY, ORDER BY, and HAVING together:

```sql
SELECT
  departamento, SUM(salario), COUNT(id)
FROM empleados
INNER JOIN departamentos ON empleados.dept_id = departamentos.id
WHERE activo = true
GROUP BY departamento
HAVING SUM(salario) > 100000
ORDER BY departamento ASC
```

<Steps>
  <Step title="Start with SELECT">
    Place a SELECT block and add your Expression and aggregate blocks.
  </Step>
  <Step title="Add FROM">
    Connect a FROM simple block. Right-click it to upgrade to FROM with JOINs if needed.
  </Step>
  <Step title="Add JOIN blocks">
    Drag JOIN blocks into the FROM block's JOINS slot. Each auto-inserts a comparison block for the ON condition.
  </Step>
  <Step title="Add WHERE">
    Connect a WHERE block. Attach a Comparison, Membership, or Quantified Comparison block into its condition input.
  </Step>
  <Step title="Add GROUP BY">
    Connect a GROUP BY block. A column block is auto-inserted; add more by chaining them.
  </Step>
  <Step title="Add HAVING">
    Connect a HAVING block after GROUP BY. Insert a `_having` aggregate block inside a Comparison block, then connect that to HAVING.
  </Step>
  <Step title="Add ORDER BY">
    Connect an ORDER BY block last. The direction dropdown appears automatically on the connected expression blocks.
  </Step>
</Steps>
